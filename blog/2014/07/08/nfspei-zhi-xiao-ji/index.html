NFS配置小记

Jul 8th, 2014 | Comments

自从买了Mac以后，开发环境渐渐从我那可怜的satellite l600（Deepin）转移到我的Mac上来，但是毕竟做得的Linux桌面开发，如果代码放到Mac上面，在我的Deepin上进行测试就比较麻烦，于是我就想到了这么几个方法：1、git在两个机器上面互相推，但是每次修改一点点做测试就得push，过于麻烦；2、rsync同步，这个比较简单，除第一次比较慢以外，后面的同步都会很快，再加上局域网，这个比第一个好一点，但是还是略麻烦；3、sshFS；4、NFS，后面两者的想法都是一样的：使用这两个基于网络的文件系统，在Deepin上面部署好了以后，Mac端只需要挂载一下就可以无缝使用。先想到的是sshfs，Mac下面配置比较简单就不多说了，完事以后，用ST写代码有明显卡顿的情况，网上搜了一下，和我遇到同样问题的人还挺多，StackOverflow上面给出的解决办法是在ST的配置文件里面加一句话’“atomic_save”: false’，这就有点毁三观了，看字面意思是ST默认不做缓存，每次编辑都会做真实IO操作？（不想深究，从Emacs转过来，就是图了个省心。。。不过知道的同学可以告诉我到底怎么回事）设置完以后情况有所好转，但是卡顿情况仍然存在。我就想这会不会是ssh每次都加密解密造成的，去掉这些多余的步骤会不会好一点？于是乎NFS就登场了。

以前并没有玩儿过NFS，之所以想到了是因为有段时间对文件系统的实现比较好奇，乱七八糟的资料看了一些，还算有所了解。不过这玩意儿用起来还挺方便，安装配置具体步骤参见Ubuntu Linux NFS Server installation and Configuration，动手配置、挂载、打开一气呵成，哈哈，等一下……没！法！编！辑！:(  只好找谷歌，过了半个来小时，才知道原委：原来NFS的权限控制是靠uid映射来完成的，像我这样的/etc/exports配置：

/home/hualet/project hualetdeMacBook-Air.local(rw, async, no_subtree_check)

也只有在Mac上新建一个用户uid为1000才能读写NFS导出的文件，又上网搜了一下（其实man exports文档写得很详细，不看文档害死人啊），发现这样的配置还算比较靠谱：

/home/hualet/project hualetdeMacBook-Air.local(rw, async, no_subtree_check, all_squash, anonuid=1000)

其中all_squash的意思是所有的用户都会被当做匿名用户对待，然后anonuid=1000就很明显是所有的匿名用户都映射到uid为1000的用户，这样我的Mac就可以顺利访问到Deepin的代码库了。

到这里NFS配置算是告一段落了，但是卡顿的问题却依然存在！一直没有搞明白原因，直到昨天跟我司运维闲聊吐槽公司网络的时候，才得知我一直使用的WiFi是被限速最严重的一个，他们一直使用着另一个优哉游哉……回去一试，Holy shit!!!
